# 开放知识格式（OKF）正式发布
**作者：**
- 萨姆·麦克维蒂（Sam McVeety）——谷歌云数据云工程部数据分析技术负责人
- 阿米尔·霍尔马蒂（Amir Hormati）——谷歌云数据云工程部BigQuery技术负责人

**发布日期：2026年6月13日**

随着基础大模型能力持续迭代，缺乏相关上下文信息往往会限制模型的实际能力，在搭建智能自主代理系统时这一问题尤为突出。尽管大模型能够辅助编写代码、总结文档、分析数据集，但想要产出精准、可落地的结果，模型仍需获取匹配场景的有效信息。

正因如此，我们今日正式推出**开放知识格式（Open Knowledge Format，OKF）**。这套开放规范将「大语言模型知识库（LLM-wiki）」范式标准化，形成一套可移植、互通互用的通用格式。该标准不绑定任何厂商，同时兼顾智能代理与人类阅读需求，专门用于承载现代AI系统所需的元数据、上下文素材与人工梳理知识库。

本次发布的 **OKF v0.1** 版本，以目录形式存储Markdown文件（文件头部附带YAML前置元数据），配套一套统一通用的基础规范。依托这套规范，不同平台产出的知识库，无需额外转换即可被各类智能代理读取使用。

整套格式设计极简：无需复杂压缩算法、无需全新运行时环境、不强制依赖任何软件开发工具包（SDK）。一份OKF知识库文件包具备三大特性：
1. **纯Markdown文本**：任意编辑器均可打开阅读，支持GitHub渲染，兼容所有检索工具建立索引；
2. **纯文件结构**：可打包为压缩包传输、托管在任意代码仓库、挂载至任意文件系统；
3. **仅YAML前置元数据**：少量结构化查询字段统一放在文件头部，包含：类型、标题、描述、资源链接、标签、时间戳。

如果你使用过Obsidian、Notion、Hugo，或是过去一年各类流行的大模型知识库工具，会对这套文件结构十分熟悉。OKF将这些零散工具共用的文件规范统一标准化，实现跨工具互通。

下文将介绍OKF能够解决企业面临的哪些痛点、底层工作原理、上手使用方法，以及该标准后续发展规划。

## 一、当前企业上下文信息体系碎片化难题
绝大多数企业中，大模型依赖的参考信息均为内部私有知识，例如：数据表结构、业务指标释义、故障处理手册、多系统关联查询逻辑、旧版API停用公告等。

现阶段，这些零散知识分散在大量互不兼容的系统中：
- 各厂商自研元数据目录，配套专属API；
- 内部知识库、第三方协作平台、共享云盘文档；
- 代码注释、函数文档字符串、数据分析单元格；
- 仅存储在资深工程师脑中的隐性经验。

当AI智能代理需要回答「如何从事件数据流统计周活跃用户？」这类问题时，必须从大量分散、互不兼容的数据源中拼凑答案。每家厂商都推出自有数据目录、专属SDK、独立知识图谱架构，知识库内容很难跨产品、跨企业迁移复用。

最终造成多重资源浪费：
所有开发智能代理的团队都要从零搭建上下文信息整合逻辑；各家数据目录厂商重复设计同类数据模型；知识库内容被锁死在产出对应的平台内，无法流通共享。

## 二、知识库：可动态更新的轻量化文档库
开发团队搭建AI智能代理的思路正在转变。以往模型需要反复检索海量文档提取固定信息；如今我们可以为智能代理搭建一套共享Markdown知识库，内容会随业务持续迭代完善。
智能代理可自主完成文档阅读、更新维护等机械工作，团队只需专注内容梳理，像管理代码一样管控知识库版本。

知名AI研究者、教育学者安德烈·卡帕西（Andrej Karpathy）在他分享的《LLM知识库》代码片段中，清晰阐述了这一思路：「大语言模型不会感到枯燥、不会忘记更新交叉引用，单次运行就能批量修改15份文档。」人类维护个人知识库时最头疼的台账更新工作，恰恰是大模型的强项。

市面上已经涌现大量同类「文档即知识库」方案，只是命名与实现各不相同：对接代码智能代理的Obsidian知识库、AGENTS.md/CLAUDE.md系列规范文件、存放index.md与日志文档的代码仓库（代理执行任务前会读取库内文档）、数据团队内部「元数据即代码」仓库等。

这套设计思路实用且强大，但所有方案均为定制化实现。卡帕西的知识库、企业内部文档库、厂商导出目录，虽然都采用Markdown+头部元数据+内部跳转链接的形式，但彼此并未做互通适配。行业没有统一标准规定每份文档必须包含哪些字段、不同文件名对应何种用途。
由此导致知识库内容只能局限在原始使用团队内部，搭建新智能代理时必须重复整理知识，产生大量冗余工作。

## 三、行业缺少的是统一格式，而非全新服务
解决上述问题，并不需要新增一套独立知识管理服务，而是需要一套通用**数据格式**，满足以下要求：
1. 无SDK门槛，任何主体都能生成知识库文件；
2. 无需复杂集成，任意工具均可读取解析；
3. 支持跨系统、跨企业、跨工具流转；
4. 可与对应业务代码一同存入版本控制系统；
5. 人类可读、代理可解析，同一份文件无需额外转换层。

OKF正是为满足以上需求设计的通用格式标准。

## 四、OKF工作原理：极简架构一览
一份OKF**知识库包**由多个Markdown文件组成，每份文件对应一个**业务概念**：数据表、数据集、业务指标、操作手册、故障预案、接口文档等均可作为独立概念文档，单文件对应单一概念，文件路径即为该概念的唯一标识。

目录结构示例：
```
sales/
├── index.md
├── datasets/
│   ├── index.md
│   └── orders_db.md
├── tables/
│   ├── index.md
│   ├── orders.md
│   └── customers.md
└── metrics/
    ├── index.md
    └── weekly_active_users.md
```

每份概念文档顶部配置简短YAML前置元数据（用于结构化检索），文档主体为自由编写的Markdown正文：
```yaml
---
type: BigQuery数据表
title: 订单表
description: 每一行对应一条已完成的客户订单
resource: https://console.cloud.google.com/bigquery?p=acme&d=sales&t=orders
tags: [销售, 营收]
timestamp: 2026-05-28T14:30:00Z
---

# 数据表结构
| 字段名        | 数据类型  | 字段说明                                  |
|---------------|-----------|-------------------------------------------|
| `order_id`    | 字符串    | 全局唯一订单编号                          |
| `customer_id` | 字符串    | 关联客户表外键，详见[客户表](/tables/customers.md) |

# 关联关系
通过`customer_id`字段与[客户表](/tables/customers.md)关联查询
```

概念之间通过标准Markdown链接互相引用，整套目录形成完整知识关联图谱，关联逻辑远比文件目录自带的父子层级关系更丰富。知识库包可选择性配置index.md（供智能代理逐级浏览层级目录）、log.md（记录全量变更时间线）。

OKF v0.1完整规范（包含兼容判定标准、跨文档链接规则、预留专用文件名清单）内容精简，单页文档即可完整承载。

### 格式设计三大核心原则
1. **最低限度强制规范**
OKF仅强制所有概念文档必须包含`type`（类型）字段；其余内容（自定义分类、拓展字段、正文板块）完全由文档创建方自由定义。规范仅统一跨工具互通所需基础字段，不限制业务内容结构。

2. **文档生成方与读取方完全解耦**
清晰区分知识库编写主体与使用主体：人工手写的知识库可直接供给AI代理读取；元数据同步流水线生成的文档可在可视化工具内浏览；一款大模型生成的知识库，能直接被另一款大模型检索调用。格式本身是统一交互契约，两端配套工具可独立替换、互不绑定。

3. **标准格式，而非封闭平台**
OKF不绑定任何云厂商、数据库、大模型服务商、智能代理开发框架。读写、部署整套格式完全不需要专属账号或私有SDK。我们选择开源发布这套标准——知识格式的价值，取决于有多少生态方兼容使用，而非由单一厂商独占。

## 配套发布工具与示例素材
为方便开发者落地使用，我们同步发布生成端、读取端两套参考实现工具：
1. **知识库生成智能代理**
自动遍历BigQuery数据集，为每张数据表、视图生成OKF概念文档；再通过一轮大模型调用抓取官方权威文档，自动补充字段说明、关联查询逻辑、引用溯源信息，完善知识库内容。
2. **静态HTML可视化工具**
无需后端服务、本地无需安装程序、数据不会外传，单文件即可将任意OKF知识库包渲染成交互式知识关联图谱。
3. **三套完整可浏览示例知识库**
全部由上述生成代理产出，已提交代码仓库作为标准兼容OKF范例：
- Google Analytics 4电商数据集：https://developers.google.com/analytics/bigquery/web-ecommerce-demo-dataset
- 栈溢出（Stack Overflow）公开数据集：https://console.cloud.google.com/bigquery?ws=!1m4!1m3!3m2!1sbigquery-public-data!2sstackoverflow
- 比特币区块链公开数据集：https://cloud.google.com/blog/topics/public-datasets/bitcoin-in-bigquery-blockchain-analytics-on-public-data?e=48754805

以上工具仅作为概念验证Demo：生成代理仅代表其中一种OKF构建方案，格式本身不限定代理框架、大模型选型；可视化工具仅提供一种读取展示方式，标准不强制依赖HTML或图谱视图。我们期待并欢迎生态内涌现更多自研生成、解析工具。

## 后续发展规划
OKF v0.1仅为起步版本，并非最终定稿标准。随着更多企业、工具接入使用，结合行业落地实践总结智能代理真实的知识检索需求，这套格式会持续迭代完善。

项目从诞生之初便完全开源开放，无论你正在搭建知识目录、数据同步流水线、AI专用知识库，或是其他AI知识相关产品，都可以基于该标准开发适配。

我们推荐开发者参与以下共建工作：
1. 阅读精简版OKF官方规范文档；
2. 针对自有业务数据源、数据库、文档平台，开发OKF生成工具；
3. 开发解析读取工具：文档浏览器、检索索引、基于知识库推理的智能代理；
4. 使用参考工具，基于自身业务数据生成OKF知识库；
5. 在代码仓库提交问题反馈、提交合并代码、拓展标准规范——规范自带版本管理，所有迭代均保障向下兼容。

这套统一格式是本次开源的核心贡献；配套工具仅用于降低落地门槛、快速验证方案。无论企业当前采用何种知识存储方案，OKF都将成为未来跨平台知识流通的通用语言。
